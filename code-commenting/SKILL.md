# 代码注释

- 只为不显而易见的业务规则、边界条件、并发控制、权限判断、时间计算或第三方协议添加注释。
- 不注释 getter、setter、简单赋值、普通 CRUD 或代码本身已经表达清楚的内容。
- 优先修正或删除错误注释，不新增重复的同义注释。
- 说明“为什么这样做”，不逐行翻译“代码做了什么”。
- Java 公共类和公共方法使用简洁 Javadoc；HTTP 接口优先补 OpenAPI 注解。
- 每个模块或文件开头用 1～3 句说明它负责什么、服务谁、主要输入或输出是什么。
- 关键类和关键方法入口用一句话说明业务职责；模块内部的小方法不重复解释。
- 遵循项目既有语言和风格；中文项目使用中文。
- 不因补注释新增类、重构业务逻辑或改变接口行为。
- 用户要求“加注释”时，先检查目标文件，只修改必要位置。
- 用户要求“清理注释”时，删除冗余、过期、重复或与代码不一致的注释。
- 最终只列出修改文件和关键注释，不写长篇解释。

## 可维护性与定位

- 每个业务模块只保留一个清晰入口；注释注明入口类、核心方法和数据落点。
- 业务规则优先写在最接近该业务的 Service 或模块中，不在多层工具类、包装类之间跳转。
- 没有至少两个真实调用方时，不新增通用工具类、适配器、包装器或抽象接口。
- Controller → Service → Mapper 的调用链保持直接；不要为一次性逻辑再增加中间层。
- 一项业务状态只保留一个修改位置；注释说明该位置是状态更新的唯一入口。
- 修改已有功能时，优先在已有业务入口扩展，不复制相同流程创建平行实现。
- 如确实需要拆分，模块说明必须列出“负责什么”和“改什么该找哪里”。

示例：

```java
/**
 * 用户优惠券服务：领取、查询和状态校验都从这里进入。
 * 修改优惠券有效期或领取规则时，优先修改 receiveCoupon。
 */
public class MobileClientApplicationServiceImpl {
}
```

## 注释优先级

1. 安全、权限、数据一致性和并发
2. 模块、类和关键方法的职责说明
3. 复杂业务状态与时间规则
4. 外部接口、签名和回调
5. 兼容性与历史决策
6. 公共 API 使用方式

## 示例

推荐：

```java
/**
 * 司机订单抢单模块：只允许待接单订单被一位司机原子领取。
 */

// 条件更新确保并发抢单时只有一位司机能成功领取订单。
return mapper.claimAvailableOrder(orderId, driverId, now) == 1;
```

避免：

```java
// 设置司机 ID
order.setDriverId(driverId);
```
