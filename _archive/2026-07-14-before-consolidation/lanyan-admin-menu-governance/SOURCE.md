# LanYan 后台菜单治理技能

本技能用于处理 LanYan 项目中的后台业务页面治理工作，目标是把代码生成产出的 CRUD 后台页收敛为符合业务职责的管理页、浏览页和公开接口，并保证菜单权限、前后端展示、接口分组规则保持一致。

## 1. 适用场景

在以下场景中优先调用本技能：

- 用户要求“整理后台菜单”“规整一级菜单”“将低频菜单收回系统工具”
- 用户要求将某些后台页面改为只读，仅保留查询和详情
- 用户要求为订单类、资金类后台页补充远程用户筛选
- 用户要求补齐订单详情、洗衣卡权益商品等关联展示模块
- 用户要求拆分后台管理接口、用户态接口、公共匿名接口
- 用户要求把本次治理规范沉淀为标准化执行文档

## 2. 目标原则

执行时必须同时满足以下原则：

- 只动业务扩展层，不修改原项目框架原生代码、核心设定和底层机制
- 页面权限、菜单权限、后端权限、接口分组必须四处一致
- 优先做“最小改动”，禁止顺手重构无关逻辑
- 后台浏览页必须真正只读，不能只在前端隐藏按钮而后端仍保留修改入口
- 小程序端或用户端接口不能直接承担后台 CRUD 语义

## 3. 页面治理规则

### 3.1 只读页判定

当页面属于“外部菜单浏览页”“低频查看页”“资产/记录查看页”时，统一按只读页治理：

- 保留：`搜索`、`重置`、`分页`、`详情`
- 移除：`新增`、`修改`、`删除`、`导出`、`状态切换`、`批量操作`
- 清理前端：表单弹窗、提交方法、校验规则、未使用的 `add/update/del/export` 调用链
- 清理后端：控制器中的 `POST/PUT/DELETE` 与只读页不一致的权限注解
- 清理菜单 SQL：去掉 `add/edit/remove/export` 按钮权限，仅保留 `list/query`

### 3.2 详情展示要求

只读页必须具备“详情”能力，推荐用 `el-dialog` 或 `el-drawer` 展示：

- 详情接口优先复用已有 `getInfo`
- 如果页面存在关联数据，则详情接口必须返回聚合数据
- 详情弹窗中优先合并展示强关联字段，避免无意义铺满表格
- 空列表必须显示明确空状态，禁止直接渲染报错

## 4. 关联展示规则

### 4.1 订单页

订单后台页若要求查看完整交易结构，详情中至少应包含：

- 订单主信息
- 订单明细列表 `orderItemList`
- 订单附加项记录列表 `orderAddonRecordList`

实现要求：

- VO 中补充聚合字段
- Service 层负责装配关联数据
- 后台控制器详情接口必须走聚合查询，而不是基础单表查询
- 前端详情弹窗中分块渲染，避免把附加项和主信息混在一列

### 4.2 用户洗衣卡页

用户洗衣卡后台页若用于查看权益使用情况，详情中至少应包含：

- 洗衣卡主记录
- 已绑定权益商品列表 `benefitProductList`
- 次数、状态、有效期等核心字段

实现要求：

- `BusUserWashCardVo` 增加权益商品列表字段
- Service 层根据 `userWashCardId` 聚合权益商品
- 前端详情弹窗中以表格或描述列表展示

## 5. 远程用户筛选规则

适用于订单类、资金类页面，例如：

- 订单
- 售后
- 洗衣卡购买记录
- 佣金明细
- 提现记录

统一做法：

- 提供后台远程用户搜索接口，例如 `/system/busUserProfile/remoteList`
- 支持按用户 ID、昵称、姓名、手机号等基础信息模糊匹配
- 前端使用远程 `el-select`，将选中的 `userId` 或关联用户字段回填到查询条件
- 必须兼容现有分页和排序逻辑，不能因为远程筛选破坏列表总数或排序行为

推荐实现口径：

- Service 层用 `searchValue` 统一接收远程搜索词
- 控制器只负责返回精简用户选项，不返回无关字段
- 订单类页面和资金类页面分别支持主用户、来源用户等不同角色筛选时，字段命名要清晰区分

## 6. 接口分组隔离规则

商品、洗衣卡、优惠券三类资源必须严格分组：

### 6.1 后台管理接口

- 路由前缀：`/system/**`
- 用途：后台授权账号的新增、修改、删除、管理查询
- 权限：必须绑定后台权限标识

### 6.2 公共匿名接口

- 路由前缀：`/public/**`
- 用途：未登录用户可访问的基础公开信息
- 数据要求：仅返回公开字段，且必须过滤下架、停用、删除数据

### 6.3 用户态接口

- 路由前缀：`/api/**`
- 用途：用户端业务消费接口
- 约束：不得继续承载后台管理 CRUD 语义
- 如果保留只读兼容接口，查询逻辑也必须走公开过滤口径或严格收窄返回范围

## 7. 菜单与权限联动清单

每次治理完成后，必须逐项核对：

- 页面模板里是否仍残留新增、修改、删除按钮
- 前端脚本里是否仍残留 `handleAdd`、`handleUpdate`、`handleDelete`、`submitForm`
- API 文件里是否仍导出只读页不再使用的增删改接口
- 后端控制器是否仍保留 `@PostMapping`、`@PutMapping`、`@DeleteMapping`
- 菜单 SQL 是否仍残留 `add/edit/remove/export` 权限按钮
- 后台详情接口是否已走聚合查询
- 远程用户筛选是否已接入并可用
- `/system`、`/api`、`/public` 是否职责清晰

## 8. 推荐执行顺序

建议按以下顺序执行：

1. 先盘点目标页面、控制器、API 文件、菜单 SQL
2. 先收敛只读页，再补详情弹窗
3. 再补订单、洗衣卡等关联展示
4. 再补远程用户筛选
5. 最后做接口分组隔离与 SQL/权限收口
6. 完成后统一做编译、诊断、页面行为验证

## 9. 验证清单

至少完成以下验证：

- 后端编译通过：`mvn -pl lanyan-admin -am -DskipTests compile`
- 前端改动文件无诊断错误
- 目标只读页不再出现 CRUD 入口
- 订单详情能看到订单明细与附加项
- 用户洗衣卡详情能看到绑定权益商品
- 订单类与资金类页面的远程用户筛选可用
- 匿名 `/public` 接口可取公开数据
- `/system` 仍保留后台管理能力

## 10. 风险处理

如果执行过程中发现以下情况，必须立即停止并汇报：

- 同一目标文件在执行期间出现新的外部改动
- 当前磁盘内容与刚读取的内容明显不一致
- 编译错误来自未知脚本批量生成结果，无法确认是否覆盖用户变更
- 脏工作区范围过大，已经无法区分本次改动与历史改动边界

处理原则：

- 以当前磁盘为准前，先明确用户是否允许继续
- 不得擅自回滚用户已有改动
- 只在明确授权后，才可在脏工作区上做最小补丁
