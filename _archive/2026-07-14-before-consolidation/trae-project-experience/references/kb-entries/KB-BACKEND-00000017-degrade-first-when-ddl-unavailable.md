# 无 DDL 权限时新字段能力先止血降级再补独立升级 SQL

## 1. 问题描述

- 在真实项目里，经常会先上线代码、后补数据库字段，或者测试环境、生产环境的建表节奏不一致。
- 当代码已经引入新字段能力，例如：
  - 推荐标记
  - 扩展状态位
  - 展示控制字段
  - 运营标记字段
- 但运行库尚未补字段时，常见结果是：
  - 列表查询直接 `Unknown column`
  - 后台切换开关时报 SQL 异常
  - 公共查询或用户查询整体不可用

## 2. 触发条件与影响范围

- 触发条件：
  - 当前环境无 DDL 权限，或 DDL 需要人工审批后才能执行。
  - 新能力依赖真实持久化字段。
  - 代码已先行上线或需先联调。
- 影响范围：
  - 查询接口
  - 写入接口
  - 后台运营开关
  - 初始化 SQL 与升级脚本交付

## 3. 复现步骤

1. 在代码中新增一个持久化字段能力，例如推荐标记或扩展状态位。
2. 不同步运行库字段，直接启动并访问列表或执行写操作。
3. 观察接口在查询、筛选、更新时出现 `Unknown column` 或类似 SQL 异常。

## 4. 根因分析

- 根因结论：
  - 代码能力和运行库结构不同步，但代码路径又强依赖新字段存在。
- 直接原因：
  - 查询、更新、筛选逻辑对新字段做了无条件使用。
- 深层原因：
  - 缺少“字段未升级时的降级策略”和“独立升级脚本交付”双保险。
- 为什么此前没有暴露：
  - 本地开发环境通常先执行过建表或改表，问题在共享环境才出现。

## 5. 分步处置方案

1. 先判断新字段能力是否允许临时降级

```text
优先判断：
- 查询功能是否可以在字段缺失时退化为默认值
- 写入功能是否可以在字段缺失时给出明确提示
- 是否必须阻断主链路，还是只影响新能力本身
```

注意事项：

- 不要为了保护新字段能力，把整个旧主链路一起拖死。

2. 对查询和读取先做止血降级

```text
常见策略：
- 新字段缺失时，列表查询返回默认值
- 新字段筛选返回空结果或忽略筛选
- 详情接口继续可用，不因扩展字段缺失整体失败
```

注意事项：

- 必须让降级语义可解释，不能悄悄返回错误业务含义。

3. 对写入和运营开关做显式拦截提示

```text
推荐方式：
- 只有在确实要写新字段时，才给出“请先执行升级 SQL”的明确提示
- 不要把底层 SQL 报错直接暴露给使用方
```

注意事项：

- 写入失败提示要说明缺少的是数据库升级，而不是让人误以为是业务校验失败。

4. 同步交付独立升级 SQL 和初始化 SQL

```text
至少同步两份内容：
- 初始化建表 SQL：保证新环境直接具备字段
- 单独升级 SQL：保证旧环境能人工补齐
```

注意事项：

- 升级 SQL 要独立成文件，方便 DBA 或运维单独执行。
- 初始化 SQL 和升级 SQL 不能只改其中一份。

## 6. 验证与验收标准

- 功能验证点：
  - 未升级字段时，旧主链路仍可正常使用。
  - 新字段能力在缺字段时给出明确可理解提示。
  - 执行升级 SQL 后，新字段能力正常生效。
- 回归验证点：
  - 字段补齐前后的查询、筛选、写入口径一致。
  - 不会因为降级代码导致已升级环境行为异常。
- 可执行测试用例：
  - 在缺字段环境下访问查询接口、筛选接口、写入接口各一次。
  - 执行升级 SQL 后再重复同样操作。
- 通过判定条件：
  - 缺字段时系统可止血运行，补字段后能力完整恢复，且升级路径清晰可执行。
