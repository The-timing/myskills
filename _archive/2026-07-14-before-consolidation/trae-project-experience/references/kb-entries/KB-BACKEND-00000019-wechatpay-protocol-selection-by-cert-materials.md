# 微信支付协议升级前先按证书材料与配置托管方式完成选型

## 1. 问题描述

- 在真实项目里，微信支付改造常常不是单一动作，而是同时混着三类诉求：
  - 从 V2 升级到 V3
  - 把配置从本地文件迁到数据库配置中心或动态配置表
  - 继续复用现有支付、回调、退款代码和证书材料
- 配置来源在这里必须单独明确，常见候选至少包括：
  - 本地 `properties / yml`
  - 数据库配置表或系统参数表，例如 `sys_config`
  - 独立配置中心
- 如果一上来默认“直接切 V3”，但现场实际只有旧版证书材料，例如 `apiclient_cert.p12`，通常会立刻出现以下冲突：
  - 下单签名方式和现有 SDK 不一致
  - 回调验签所需材料不齐
  - 退款链路仍依赖 p12 证书，无法无缝迁移
  - 配置既想迁到数据库，又还保留文件配置，形成双口径
- 这类问题的本质，不是“支付代码不会写”，而是协议选型、证书材料和配置来源没有先统一决策。

## 2. 触发条件与影响范围

- 触发条件：
  - 项目已存在稳定运行的微信支付 V2 链路。
  - 现场证书材料不完整，只有 `apiclient_cert.p12` 或只具备旧版商户密钥。
  - 用户要求统一动态配置，并且已经明确目标配置来源，例如收口到数据库配置表、系统参数表（如 `sys_config`）或配置中心。
- 影响范围：
  - 支付下单接口
  - 支付回调接口
  - 退款申请与退款状态推进
  - 运维部署材料与初始化 SQL

## 3. 复现步骤

1. 在已有 V2 支付项目中，提出“升级到 V3，同时把支付配置改成动态配置”的需求。
2. 在未确认现场证书材料是否齐全的情况下，直接按 V3 改造下单、回调或退款。
3. 继续沿用旧版 `p12` 证书或旧 SDK 退款能力。
4. 最终出现协议、证书和配置来源不一致，导致方案反复调整，甚至需要从 V3 回退到 V2。

## 4. 根因分析

- 根因结论：
  - 微信支付改造前没有先做“协议能力盘点”和“配置托管盘点”，导致技术路线建立在错误前提上。
- 直接原因：
  - 把“想升级 V3”误当成“已经具备 V3 所需材料”。
  - 把“想统一配置来源”误当成“必须同步更换支付协议”，没有先写清配置最终来自哪里。
- 深层原因：
  - 没有把支付协议、证书材料、SDK 能力、配置托管方式拆成四个独立决策点。
  - 没有优先保护现有支付主链路的稳定性。
- 为什么此前没有暴露：
  - 本地开发常常默认材料齐全，或只验证下单不验证退款与回调；真正落到项目改造时，材料缺口和历史链路耦合才会暴露。

## 5. 分步处置方案

1. 先做支付材料与链路盘点，再决定 V2 还是 V3

```text
盘点项至少包括：
- 当前下单协议：V2 还是 V3
- 当前回调处理：XML 还是 JSON + 平台证书验签
- 当前退款材料：是否仍依赖 apiclient_cert.p12
- 当前已有证书材料：
  - p12
  - PEM 私钥
  - 平台证书
  - APIv3Key
- 当前配置来源：
  - 本地 properties / yml
  - 数据库配置表
  - 独立配置中心
```

注意事项：

- 不要把“用户希望升级”直接等同于“现场可以升级”。
- 只要退款仍强依赖 `apiclient_cert.p12`，就必须优先评估是否继续保留 V2。
- 在进入代码改造前，必须把配置来源写成可执行结论，例如“继续读文件”“收口到 `sys_config`”“接入独立配置中心”，不能只写“改成动态配置”。

2. 如果现场只有 p12 且已有 V2 主链路，优先选择“V2 + 动态配置收口”

```text
推荐动作：
- 保留现有 V2 下单、V2 回调、V2 退款协议
- 只改配置来源，不强行改支付协议
- 统一把 appId、secret、mchId、mchKey、domainName、certPath 等参数收口到数据库配置表或配置中心
- 如果项目已经采用系统参数表，优先明确为单一配置来源，例如统一收口到 `sys_config`
```

注意事项：

- 这种方案的核心是“先收口配置，后评估升级协议”，不要两件事一次硬绑。
- 如果项目已稳定依赖某个 V2 SDK，优先保留 SDK 和回调格式，避免扩大改动面。

3. 只有在 V3 材料齐全时，才进入 V3 改造

```text
V3 至少要确认：
- 商户私钥 PEM
- 商户证书序列号
- 微信支付平台证书
- APIv3Key
- 对应 SDK 或自研签名/验签能力
```

注意事项：

- 缺任意关键材料时，不要只改下单接口后再把退款留在旧实现里长期混跑。
- 除非有明确灰度方案，否则不要在同一项目里长期混用 V2 和 V3 主链路。

4. 配置收口时同步交付初始化 SQL 或配置导入脚本

```text
至少交付：
- 配置键清单
- 默认说明
- 可直接执行的初始化 SQL 或导入脚本
- 回调地址的拼接规则
```

注意事项：

- 不能只改代码读取方式，不交付运行环境配置。
- 如果配置要从文件迁到数据库，必须给运维或实施可直接落地的初始化脚本。
- 初始化脚本里应直接体现配置来源对应的键清单和存放位置，例如数据库参数表中的 `config_key/config_value` 结构。

## 6. 验证与验收标准

- 功能验证点：
  - 支付协议、退款协议和证书材料彼此一致。
  - 配置统一从新的动态配置来源读取。
  - 下单、支付回调、退款三条链路都能闭环。
- 回归验证点：
  - 配置迁移后，旧主链路行为不改变。
  - 不会出现“下单一套协议、退款另一套协议、配置第三套来源”的三重口径分裂。
- 可执行测试用例：
  - 在只有 `apiclient_cert.p12` 的环境下，验证保留 V2 协议并切换到数据库配置后，下单、回调、退款都正常。
  - 在材料齐全的环境下，再单独评估是否切 V3，而不是与配置迁移捆绑验证。
- 通过判定条件：
  - 支付协议选型建立在真实材料和现有链路基础上，配置迁移后可直接部署，且没有引入新的协议口径冲突。
