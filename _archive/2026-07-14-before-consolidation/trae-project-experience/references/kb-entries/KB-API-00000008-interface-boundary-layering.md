# 公共态用户态后台态接口职责分层

## 1. 问题描述

- 同一系统通常会存在不同调用角色：
  - 匿名访问
  - 登录用户访问
  - 后台管理访问
- 如果接口边界没有分层，常见结果是：
  - 客户端直接调用后台接口
  - 权限校验异常
  - 接口语义越来越混乱
  - 后期难以治理

## 2. 触发条件与影响范围

- 触发条件：
  - 项目同时包含公共查询、用户消费和后台管理能力。
  - 某一能力既想被前台用，又想被后台用。
- 影响范围：
  - 接口前缀设计
  - 权限规则
  - 控制器职责划分

## 3. 复现步骤

1. 在系统中只保留一套接口供所有角色复用。
2. 让匿名用户、登录用户和后台管理员都访问同一控制器。
3. 继续新增权限、筛选和管理能力。
4. 观察接口语义、权限规则和对象结构逐步混杂。

## 4. 根因分析

- 根因结论：
  - 接口没有按调用角色和职责做分层设计。
- 直接原因：
  - 为了快速复用，把后台接口直接暴露给前台或客户端。
- 深层原因：
  - 没有明确区分“公共态”“用户态”“后台态”的边界。
- 为什么此前没有暴露：
  - 早期功能少时，一套接口还能勉强覆盖所有角色。

## 5. 分步处置方案

1. 先按调用角色划分接口边界

```text
建议至少分为：
- 公共态：匿名可读或无需业务身份即可访问
- 用户态：登录用户消费、查询、提交
- 后台态：管理、治理、审核、运维
```

注意事项：

- 接口边界按职责划分，不按“谁方便复用”划分。

2. 再决定对象复用和控制器复用策略

```text
优先顺序：
- 先复用对象
- 再复用服务层能力
- 控制器按角色拆分
```

注意事项：

- 不要为了少写一个接口，就让客户端走后台权限体系。

3. 明确每类接口的访问规则和返回语义

```text
公共态：
- 以只读、基础查询为主

用户态：
- 以用户本人业务语义为主

后台态：
- 以管理动作和治理能力为主
```

注意事项：

- 同一对象可以复用，但不同角色的语义不能混在一个接口里。

## 6. 验证与验收标准

- 功能验证点：
  - 匿名请求只能访问公共态接口。
  - 登录用户只访问用户态接口。
  - 后台管理操作不对前台暴露。
- 回归验证点：
  - 对象和服务层仍可复用，不因控制器分层导致重复实现。
- 可执行测试用例：
  - 分别用匿名、普通用户、管理员身份访问三个层次接口。
- 通过判定条件：
  - 权限边界清晰，接口语义清晰，调用角色与接口职责一致。
