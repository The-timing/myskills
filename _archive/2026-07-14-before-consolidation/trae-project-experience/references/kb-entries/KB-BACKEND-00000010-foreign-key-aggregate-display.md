# 外键字段优先后端聚合回显业务展示信息

## 1. 问题描述

- 许多接口会返回外键字段，例如用户 ID、商品 ID、组织 ID、来源对象 ID。
- 如果接口只返回裸 ID，前端常常需要额外请求其它接口拼装展示信息，导致：
  - 页面逻辑变复杂
  - 多次请求影响性能
  - 列表和详情展示口径不一致

## 2. 触发条件与影响范围

- 触发条件：
  - 返回对象存在外键字段。
  - 页面展示需要人类可读信息而非 ID。
  - 同一外键会在列表、详情、筛选等多个地方展示。
- 影响范围：
  - 返回对象设计
  - 服务层聚合逻辑
  - 前端渲染复杂度

## 3. 复现步骤

1. 让后端接口只返回多个外键 ID。
2. 前端在列表和详情中分别调用其它接口补展示信息。
3. 观察前端出现多处拼装逻辑、额外请求和口径不一致问题。

## 4. 根因分析

- 根因结论：
  - 用于展示的业务信息没有在后端聚合层一次性回显。
- 直接原因：
  - 接口设计只关注数据表结构，没有关注消费方展示语义。
- 深层原因：
  - 把“可展示数据准备”错误地推给了前端。
- 为什么此前没有暴露：
  - 早期只有少量字段时，前端临时拼接还能勉强维持。

## 5. 分步处置方案

1. 在返回对象中增加展示型扩展字段

```text
例如：
- userName
- userPhone
- productTitle
- organizationName
```

注意事项：

- 展示字段与主键字段并存，不要删除原外键。
- 展示字段命名要稳定、语义清晰。

2. 在服务层统一做聚合注入

```text
推荐方式：
- 批量查询关联对象
- 构建 ID -> 展示信息 映射
- 一次性回填到返回对象
```

注意事项：

- 优先批量处理，避免逐条查询导致 N+1 问题。
- 列表和详情尽量共用同一套聚合逻辑。

3. 前端只消费聚合后的展示字段

```text
前端展示优先读取：
- 展示型扩展字段
- 裸 ID 仅用于跳转、提交、筛选
```

注意事项：

- 不要让前端再次承担主展示信息的拼装责任。

## 6. 验证与验收标准

- 功能验证点：
  - 列表和详情能直接展示业务名称、手机号、摘要等信息。
  - 前端不需要额外请求即可完成主要展示。
- 回归验证点：
  - 原外键字段仍可用于提交和查询。
  - 列表和详情的展示口径一致。
- 可执行测试用例：
  - 同时校验列表接口和详情接口的聚合字段是否齐全。
- 通过判定条件：
  - 主要展示信息由后端一次性回显，前端不再做重复拼装。
