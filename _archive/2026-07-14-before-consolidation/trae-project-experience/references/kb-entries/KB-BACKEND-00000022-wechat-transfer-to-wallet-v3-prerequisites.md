# 商家提现到微信零钱应按微信支付 V3 商家转账链路建设前置条件与验签

## 1. 问题描述

- 在小程序或公众号业务中，用户提现到微信零钱不是“退款接口”的变体，而是微信支付 V3 的商家转账能力。
- 这类需求经常和微信退款、企业付款到零钱、红包发放混在一起，导致实现时出现口径错位：
  - 把 V3 商家转账误当成 V2 退款
  - 缺少商户证书序列号却直接调接口
  - 有请求响应但没做平台证书验签
  - 金额仍按元传递，没有先转分
- 这类问题的本质不是“接口不会调”，而是没有先把微信提现到零钱的协议类型、前置材料和验签动作定义清楚。

## 2. 触发条件与影响范围

- 触发条件：
  - 系统存在用户提现审核通过后打款到微信零钱的业务。
  - 开发准备使用 IJPay 的 `WxPayApi.v3(...)` + `TransferApiEnum.TRANSFER_BATCHES`。
  - 现场需要同时管理商户号、商户证书序列号、平台证书、公钥等配置。
- 影响范围：
  - 提现审核通过后的出款动作
  - 商户转账批次创建
  - 微信 V3 签名与响应验签
  - 配置托管与部署材料

## 3. 复现步骤

1. 用户提现审核通过后，需要把金额打到 `openid` 对应的微信零钱。
2. 直接复用 V2 支付或退款心智实现，而没有先确认这是 V3 商家转账场景。
3. 忽略证书序列号、公钥或平台证书验签要求。
4. 最终出现接口调用失败、签名校验失败、金额错误或实现方案反复调整。

## 4. 根因分析

- 根因结论：
  - 商家提现到微信零钱本质上是微信支付 V3 的独立链路，不能用 V2 退款思路硬套。
- 直接原因：
  - 没有把“出款协议”“请求签名材料”“响应验签材料”“金额单位”拆开确认。
- 深层原因：
  - 团队容易把“钱退给用户”和“钱打到零钱”混成同一个问题。
  - 只关注接口路径，没有同时确认配置来源和部署所需的密钥材料。
- 为什么此前没有暴露：
  - 很多项目一开始只做收款，真正接入提现时才暴露 V3 商家转账的材料要求。

## 5. 分步处置方案

1. 先确认这是“微信提现到零钱”，不是退款

```text
判断口径：
- 退款：针对原支付单退回，属于支付逆向流程
- 提现到零钱：平台向用户微信零钱主动打款，属于商家转账流程
```

注意事项：

- 不要把售后退款代码直接改成提现代码。
- 不要复用 V2 退款证书链路处理微信提现。

2. 统一按 V3 商家转账接口准备请求

```java
IJPayHttpResponse response = WxPayApi.v3(
    RequestMethodEnum.POST,
    WxDomainEnum.CHINA.toString(),
    TransferApiEnum.TRANSFER_BATCHES.toString(),
    mchId,
    merchantSerialNo,
    nonceStr,
    privateKeyOrPublicKeyConfig,
    JSONUtil.toJsonStr(batchTransferModel)
);
```

注意事项：

- 请求体应使用 `BatchTransferModel` 和 `TransferDetailInput` 这类明确的 V3 模型。
- 批次号、明细单号应保证唯一，避免重复打款。
- 金额必须先从元转换为分，再传给微信。

3. 响应后必须做平台证书验签

```java
boolean verifySignature = WxPayKit.verifySignature(response, platformCert);
```

注意事项：

- 不能只看 HTTP 200 就认定打款成功。
- 平台证书验签不通过时，应视为响应不可直接信任。
- 日志中应记录响应摘要和验签结果，但不要泄露敏感密钥。

4. 配置按职责分层存放

```text
推荐分层：
- application.yml：工程静态行为
- sys_config：微信商户号、appId、商户证书序列号、公钥、平台证书等第三方接入参数
- bus_setting：业务提现规则，如最低提现金额、审核开关、手续费口径
```

注意事项：

- 不要把 V3 接入密钥材料混到业务设置里。
- 不要同时在多个地方维护同一套商户号/证书配置。

5. 提现链路要明确区分“受理成功”和“最终到账”

```text
建议状态拆分：
- 审核通过
- 发起微信转账请求成功
- 微信受理成功 / 待最终结果
- 最终到账成功
- 打款失败 / 待人工处理
```

注意事项：

- 微信受理成功不一定等于用户已经到账。
- 需要结合批次查询或回调机制做最终状态收敛。

## 6. 验证与验收标准

- 功能验证点：
  - 提现审核通过后能正常创建微信商家转账批次。
  - 金额传值单位正确，按分提交。
  - 响应验签通过后才进入后续业务确认。
- 回归验证点：
  - 不影响现有 V2 收款、回调、退款链路。
  - 提现配置和退款配置边界清晰，不互相污染。
- 可执行测试用例：
  - 使用 1 笔小额提现验证请求能成功受理。
  - 模拟验签失败，确认系统不会直接把失败响应当成成功。
  - 校验批次号、明细号重复时系统是否有幂等保护。
- 通过判定条件：
  - 提现到零钱链路明确按 V3 商家转账实现，配置清晰、金额正确、验签闭环完整。
