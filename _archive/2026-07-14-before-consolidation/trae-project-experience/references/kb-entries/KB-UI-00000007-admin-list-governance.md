# 后台业务列表优先采用状态主筛选聚合列与详情弹窗

## 1. 问题描述

- 后台业务列表随着字段增长，容易出现列过多、页面过宽、筛选分散、详情信息难展示的问题。
- 如果继续在原表格中平铺字段，最终会导致：
  - 主信息淹没在长表格中
  - 强主维度筛选难以操作
  - 查看详情必须频繁横向滚动
  - 操作按钮拥挤且难维护

## 2. 触发条件与影响范围

- 触发条件：
  - 列表页字段数量持续增长。
  - 存在状态、用户、关联对象等高频筛选条件。
  - 详情信息无法在列表中完整表达。
- 影响范围：
  - 页面布局
  - 查询效率
  - 用户操作效率

## 3. 复现步骤

1. 在后台列表页持续平铺新增字段和按钮。
2. 将状态筛选、关联对象筛选都塞进普通搜索区。
3. 让用户查看完整详情或执行多个操作。
4. 观察页面宽度、信息密度和交互成本不断上升。

## 4. 根因分析

- 根因结论：
  - 列表页没有按“主筛选 + 聚合摘要 + 详情分层”的思路治理。
- 直接原因：
  - 将所有信息都试图一次性平铺在表格中。
- 深层原因：
  - 没有明确区分“列表职责”和“详情职责”。
- 为什么此前没有暴露：
  - 早期字段少时，平铺式表格还能勉强承载。

## 5. 分步处置方案

1. 将主状态筛选提升为顶部主筛选组件

```text
推荐：
- 状态 tabs / segmented control
- “全部”使用显式值
```

注意事项：

- 主筛选不要和低频搜索条件混在一个下拉区。

2. 将强关联字段改为聚合列

```text
推荐聚合方式：
- 基础信息块
- 用户与关联对象块
- 金额与状态块
- 时间信息块
```

注意事项：

- 聚合列内信息应按主次排序，不要只是换行堆砌。

3. 将明细信息移入详情弹窗或抽屉

```text
推荐：
- dialog
- drawer
- descriptions + 子表格
```

注意事项：

- 列表页只保留决策所需摘要，不要强行承载完整明细。

## 6. 验证与验收标准

- 功能验证点：
  - 主状态筛选能快速切换。
  - 列表行能在有限宽度下表达核心信息。
  - 详情可在弹窗或抽屉中完整查看。
- 回归验证点：
  - 原有筛选条件和操作仍可使用。
- 可执行测试用例：
  - 对比治理前后，在相同数据量下完成一次筛选、一次定位、一次详情查看的交互步骤数量。
- 通过判定条件：
  - 列表宽度可控，主信息可读，详情不再依赖表格平铺。
