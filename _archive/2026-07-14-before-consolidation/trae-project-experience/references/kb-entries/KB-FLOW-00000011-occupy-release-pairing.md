# 资源占用类能力必须成对设计释放路径

## 1. 问题描述

- 某些业务会在创建、预览确认、提交或执行阶段占用资源。
- 如果只设计“如何占用”，不设计“何时释放”，就会出现：
  - 资源长期悬挂
  - 后续请求被错误拦截
  - 数据和真实业务状态不一致

## 2. 触发条件与影响范围

- 触发条件：
  - 系统中存在库存、优惠、额度、资格、权益等资源。
  - 创建或执行阶段会对资源做锁定、扣减、预占用。
- 影响范围：
  - 数据一致性
  - 并发处理
  - 取消与回滚流程

## 3. 复现步骤

1. 构造一个在创建或提交时会占用资源的流程。
2. 执行占用后，中途触发取消、支付失败、回滚或超时。
3. 检查资源状态，观察资源未恢复或未释放。
4. 再次请求同一资源，发现被错误判定为不可用。

## 4. 根因分析

- 根因结论：
  - 资源占用设计缺少对称的释放与补偿路径。
- 直接原因：
  - 开发时只关注成功路径，没有覆盖失败路径和中断路径。
- 深层原因：
  - 资源管理没有被建模为完整生命周期。
- 为什么此前没有暴露：
  - 正常成功路径会掩盖失败与中断分支的问题。

## 5. 分步处置方案

1. 先枚举所有资源占用点和所有释放触发点

```text
占用点：
- 创建
- 提交
- 支付前
- 支付后

释放点：
- 取消
- 失败
- 超时
- 回滚
- 补偿任务
```

注意事项：

- 不要只枚举主流程，必须覆盖异常分支和补偿分支。

2. 为每类资源建立对称释放策略

```text
推荐：
- 同事务回滚释放
- 显式取消释放
- 超时任务释放
- 幂等补偿释放
```

注意事项：

- 释放逻辑必须幂等，避免重复释放造成二次错误。

3. 将占用和释放统一纳入同一资源生命周期设计

```text
至少定义：
- 占用条件
- 占用动作
- 释放条件
- 释放动作
- 最终完成后的归档状态
```

注意事项：

- 不要把释放逻辑散落到多个地方而没有统一视图。

## 6. 验证与验收标准

- 功能验证点：
  - 正常成功路径下资源状态正确。
  - 取消、失败、回滚、超时路径下资源都能恢复。
- 回归验证点：
  - 重复执行释放逻辑不会造成异常。
- 可执行测试用例：
  - 分别验证成功、取消、失败、超时四种路径。
- 通过判定条件：
  - 任一路径结束后资源状态都与最终业务状态一致。
