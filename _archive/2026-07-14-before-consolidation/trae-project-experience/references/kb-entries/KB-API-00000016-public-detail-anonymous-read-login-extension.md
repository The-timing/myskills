# 公共详情接口采用匿名只读与登录扩展行为分离

## 1. 问题描述

- 某些公共详情接口本身允许匿名访问，例如商品详情、内容详情、视频详情、报告预览等。
- 但产品又往往希望在登录用户访问详情时，顺手执行扩展行为，例如：
  - 追加观看记录
  - 写浏览足迹
  - 标记首次访问
  - 发放首访奖励
- 如果把这些登录态动作直接混进公共读取主流程，常见结果是：
  - 未登录请求直接报错
  - 公共接口被迫依赖鉴权上下文
  - 登录扩展逻辑因为异常吞没而不稳定

## 2. 触发条件与影响范围

- 触发条件：
  - 存在可匿名访问的详情接口。
  - 又希望在已登录时自动追加写操作或奖励动作。
  - 登录用户获取方式可能在匿名请求下抛异常或返回空值。
- 影响范围：
  - 公共控制器设计
  - 登录态扩展行为的稳定性
  - 详情读取与附带写入的一致性

## 3. 复现步骤

1. 设计一个允许匿名访问的详情接口。
2. 在该接口主流程中直接执行依赖当前登录用户的写入逻辑。
3. 用匿名请求访问该接口，观察接口报错或被迫改成需要登录。
4. 再用已登录请求访问，观察详情读取和附带写入逻辑耦合在一起，后续难以治理。

## 4. 根因分析

- 根因结论：
  - 把“公共只读职责”和“登录态扩展行为”混在了同一个强依赖鉴权的主流程中。
- 直接原因：
  - 详情接口在读取主流程里直接依赖当前用户上下文，没有对匿名访问做兼容分支。
- 深层原因：
  - 没有把“读取基线能力”和“登录后附带行为”拆成两层语义。
- 为什么此前没有暴露：
  - 联调时通常默认使用已登录账号访问，匿名场景被忽略。

## 5. 分步处置方案

1. 先固定公共详情接口的只读基线

```text
基线原则：
- 详情对象本身可匿名读取，就先保证匿名请求能稳定返回
- 详情主流程只负责读取和组装详情数据
```

注意事项：

- 不要因为要写足迹、观看记录、积分，就把公共详情接口整体升级成必须登录。

2. 再把登录扩展行为做成可选分支

```text
推荐处理：
- 先尝试获取当前登录用户
- 获取失败或为空时：直接跳过扩展行为
- 获取成功时：再调用记录写入、首访奖励、用户态补充逻辑
```

注意事项：

- 登录扩展行为必须与详情读取解耦。
- 匿名请求不能因为扩展行为而失败。

3. 登录扩展逻辑下沉到服务层，并保持幂等

```text
推荐下沉动作：
- saveDetailVisitRecord(memberId, bizId)
- appendFootprint(memberId, bizId)
- awardFirstVisitPointsIfNecessary(memberId, bizId)
```

注意事项：

- 首访奖励、足迹写入、观看记录等动作必须幂等。
- 详情接口只负责“判断是否有登录用户 + 触发服务”，不把业务细节堆在控制器。

## 6. 验证与验收标准

- 功能验证点：
  - 匿名访问详情接口时，详情能正常返回。
  - 已登录访问详情接口时，详情返回正常且扩展行为成功执行。
- 回归验证点：
  - 匿名请求不会写入脏记录。
  - 已登录请求重复访问时，不会重复发放首访奖励。
- 可执行测试用例：
  - 用匿名身份请求详情接口一次。
  - 用登录身份首次请求详情接口一次。
  - 用同一登录身份再次请求详情接口一次。
- 通过判定条件：
  - 匿名读取稳定，登录扩展可执行，且扩展行为具备幂等性。
