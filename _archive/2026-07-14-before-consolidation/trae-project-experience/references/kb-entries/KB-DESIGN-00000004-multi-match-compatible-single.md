# 多对象匹配场景保留兼容单对象字段

## 1. 问题描述

- 某些场景下，请求会同时命中多个关联对象。
- 但历史接口通常只返回一个对象字段，导致：
  - 完整命中结果丢失
  - 前端无法展示所有候选或已命中对象
  - 为兼容旧调用方，又不能直接把原单对象字段删掉

## 2. 触发条件与影响范围

- 触发条件：
  - 一个输入会命中多个资源、多个策略、多个关联对象或多个候选结果。
  - 旧接口或旧页面依赖一个单对象字段。
- 影响范围：
  - 返回结构设计
  - 前端展示逻辑
  - 默认优先级排序规则

## 3. 复现步骤

1. 构造一个会同时命中多个对象的请求。
2. 使用仅支持单对象返回的旧接口结构。
3. 观察只有一个对象被返回，其它命中结果丢失。
4. 尝试直接改成只返回列表，观察旧调用方发生兼容问题。

## 4. 根因分析

- 根因结论：
  - 接口模型仍停留在单对象返回阶段，但实际业务已进入多对象匹配阶段。
- 直接原因：
  - 返回结构无法同时满足“完整表达结果”和“兼容旧字段”。
- 深层原因：
  - 对象优先级规则没有被显式建模。
- 为什么此前没有暴露：
  - 在早期输入简单时，系统通常只会命中一个对象。

## 5. 分步处置方案

1. 新增对象列表字段表达完整结果

```text
新增：
- matched_xxx_list
- candidate_xxx_list
- related_xxx_list
```

注意事项：

- 列表字段必须表达完整命中结果，不能只放部分数据。
- 列表元素顺序应按优先级规则稳定排序。

2. 保留原单对象字段用于兼容

```text
保留：
- current_xxx
- primary_xxx
- matched_xxx
```

注意事项：

- 单对象字段默认取排序后的第一优先对象。
- 不能让旧字段和新列表字段出现语义冲突。

3. 显式定义优先级规则

```text
优先级可由以下因素决定：
- 时间先后
- 过期时间
- 优先级权重
- 明确排序字段
```

注意事项：

- 优先级规则必须写入接口说明或知识条目。
- 不允许默认依赖数据库返回顺序。

## 6. 验证与验收标准

- 功能验证点：
  - 多命中时列表字段返回完整结果。
  - 单对象字段仍可用，且指向优先级最高的对象。
- 回归验证点：
  - 旧前端不因新增列表字段而中断。
  - 新前端可以读取完整列表。
- 可执行测试用例：
  - 构造单命中和多命中两组输入，验证两种返回结构都正确。
- 通过判定条件：
  - 完整结果不丢失，旧字段不失效，优先级规则可解释且稳定。
