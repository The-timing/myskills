# 远程筛选组件默认提供可用候选数据

## 1. 问题描述

- 页面中的远程搜索下拉如果只有在输入关键词时才触发请求，通常会在以下场景表现很差：
  - 首次进入页面时为空
  - 下拉展开时为空
  - 清空搜索词后仍为空
- 这会导致用户不知道可选内容范围，也不知道该搜什么关键词。

## 2. 触发条件与影响范围

- 触发条件：
  - 使用远程筛选或远程下拉组件。
  - 目标对象属于高频可选择实体。
  - 用户并不总是知道明确搜索词。
- 影响范围：
  - 列表筛选
  - 表单选择
  - 页面初次使用体验

## 3. 复现步骤

1. 构建一个仅支持 `remoteMethod(query)` 的远程下拉。
2. 页面初始化时不加载任何默认数据。
3. 展开下拉框或清空输入框内容。
4. 观察组件无候选项，必须凭空输入关键词才能继续。

## 4. 根因分析

- 根因结论：
  - 远程筛选组件缺少默认候选数据策略。
- 直接原因：
  - 组件把“搜索”误设计成唯一取数入口。
- 深层原因：
  - 只从实现成本考虑，没有从交互引导和高频使用场景考虑。
- 为什么此前没有暴露：
  - 开发和测试人员通常知道精确关键词，因此难以感知首次使用问题。

## 5. 分步处置方案

1. 增加默认候选加载逻辑

```text
在页面创建时主动拉取一批默认候选数据，例如最近数据、热门数据或前 N 条数据。
```

注意事项：

- 默认候选必须有限量，避免一次性加载过多数据。
- 默认候选的排序应稳定且可解释。

2. 在组件交互关键节点恢复默认候选

```text
至少覆盖：
- 页面初始化
- 下拉展开且当前无数据
- 搜索词清空后
```

注意事项：

- 清空关键词后不要直接置空，应回退到默认候选。
- 如果有缓存，可优先读缓存再回源请求。

3. 将搜索结果与默认候选分层处理

```text
默认候选：
- 用于引导用户和高频快速选择

搜索结果：
- 用于精确查找
```

注意事项：

- 不要把搜索结果永久覆盖默认候选缓存。

## 6. 验证与验收标准

- 功能验证点：
  - 页面初始化时有默认候选。
  - 下拉展开时无空白状态。
  - 清空搜索词后能恢复默认候选。
- 回归验证点：
  - 输入关键词后远程搜索仍正常工作。
- 可执行测试用例：
  - 分别测试初始化、展开、输入搜索、清空搜索词四个交互节点。
- 通过判定条件：
  - 远程组件在无关键词时仍具备可用性，且不会影响精确搜索能力。
