# 工程静态配置、第三方接入配置与业务运行时配置要按职责分层存放

## 1. 问题描述

- 很多项目在演进过程中会同时出现三类配置：
  - 随代码和环境一起发布的工程静态配置
  - 第三方平台接入参数
  - 业务运行时由运营或业务人员维护的配置
- 如果新增配置时没有先判断归属，就容易出现：
  - 把业务运行时配置写进 `application.yml`
  - 把工程级开关塞进数据库表
  - 第三方参数一部分在文件里、一部分在系统参数表里
  - 同一个配置在多个来源重复存放，最终不知道代码实际读的是哪一份
- 一旦混放，后果通常不是马上报错，而是发布、联调、灰度和运营维护阶段持续出现口径错位。

## 2. 触发条件与影响范围

- 触发条件：
  - 项目既有配置文件，又有数据库参数表或业务设置表。
  - 需求要求新增“可配置项”，但没有定义配置归属。
  - 不同角色分别维护配置，例如研发、运维、运营都可能改配置。
- 影响范围：
  - 配置读取链路
  - 第三方接入稳定性
  - 业务运行时行为
  - 发布与回滚成本

## 3. 复现步骤

1. 在一个已有 `application.yml`、系统参数表和业务设置表的项目里新增配置项。
2. 未区分配置职责，直接按“哪里方便就放哪里”的方式落配置。
3. 继续新增同类配置时，团队成员根据各自习惯把配置分散到不同来源。
4. 最终出现排查时找不到真实配置源、联调值和正式值混淆、业务运行时参数无法稳定维护等问题。

## 4. 根因分析

- 根因结论：
  - 配置治理缺少按职责划分的统一边界，导致所有配置都被当成“能读到就行”的技术实现问题。
- 直接原因：
  - 新增配置前没有先判断它是工程静态配置、第三方接入配置，还是业务运行时配置。
  - 没有规定每一类配置的唯一优先存放位置。
- 深层原因：
  - 配置来源和配置所有权没有绑定，研发、运维、运营维护边界模糊。
  - 团队更关注“先跑起来”，没有持续治理配置分层。
- 为什么此前没有暴露：
  - 早期配置项少时，混放也能运行；一旦第三方接入增多、业务动态配置增多，问题才会放大。

## 5. 分步处置方案

1. 新增配置前先做职责归类

```text
先回答三个问题：
- 这个配置是否随代码发布和环境切换一起变化？
- 这个配置是否属于第三方平台接入参数？
- 这个配置是否需要在项目运行期由业务或运营人员维护？
```

注意事项：

- 不要一上来先想“放哪张表”，先判断配置职责和所有权。

2. 工程静态配置统一放工程配置文件

```text
推荐放置：
- application.yml
- application-*.yml

典型内容：
- 端口、数据源、缓存、中间件地址
- Spring Boot 工程参数
- 只随环境发布变更、不要求业务侧运行时维护的代码级配置
```

注意事项：

- 这类配置应跟随部署环境和代码包管理，不应要求运营在业务后台手工维护。

3. 第三方接入参数统一收口到系统参数层

```text
推荐放置：
- 系统参数表 / 动态参数表
- 常见示例：sys_config

典型内容：
- 微信、短信、地图、对象存储、第三方 AI 平台等接入参数
- appId、secret、商户号、回调域名、接口地址等
```

注意事项：

- 同一类第三方参数必须收口到单一来源，避免文件和数据库双份并存。
- 如果项目已有系统参数表，优先复用，不要再额外新建散落配置入口。

4. 业务运行时配置统一收口到业务设置层

```text
推荐放置：
- 业务设置表 / 业务动态配置表
- 常见示例：bus_setting

典型内容：
- 协议文案、活动规则、积分规则、页面展示位、业务说明、运行期可调整的业务参数
- 直接参与项目逻辑判断或页面展示，且需要运营在运行时维护的配置
```

注意事项：

- 这类配置的核心特征不是“存在数据库里”，而是“项目运行期可维护且直接影响业务逻辑”。
- 不要把纯工程配置塞进业务设置表，否则会让业务后台承担不该承担的运维职责。

5. 为每类配置补唯一来源说明和初始化交付

```text
至少交付：
- 配置项归属说明
- 唯一读取来源
- 初始化 SQL / 默认值 / 配置示例
- 读取优先级说明
```

注意事项：

- 文档和代码里都要写清楚“最终读哪里”。
- 禁止长期存在“文件一份、数据库一份、代码里再默认一份”而没人知道谁优先。

## 6. 验证与验收标准

- 功能验证点：
  - 新增配置可以被准确归类到唯一来源。
  - 代码读取来源和文档说明一致。
  - 第三方参数、业务动态配置和工程静态配置不再混放。
- 回归验证点：
  - 发布环境切换时，工程静态配置仍由部署流程管理。
  - 业务侧运行时调整不需要修改 `application.yml`。
  - 第三方联调值和正式值不会因为多来源并存而混淆。
- 可执行测试用例：
  - 分别新增一个工程级参数、一个第三方接入参数、一个业务运行时参数。
  - 校验三者是否分别进入工程配置文件、系统参数表和业务设置表。
  - 重启服务并执行业务链路，确认三类配置都从预期来源读取。
- 通过判定条件：
  - 团队可以明确回答每个配置“为什么放这里、谁来维护、代码最终从哪里读”，且三类配置边界稳定。
