# 主对象摘要状态与流程对象详细状态分层

## 1. 问题描述

- 一个主对象通常既要用于列表展示、统计和筛选，又要承载完整流程状态。
- 如果直接把所有详细状态都塞进主对象，会出现：
  - 状态值过多，难以理解
  - 列表筛选口径混乱
  - 前后端对状态含义不一致
  - 明细流程和概要展示互相耦合

## 2. 触发条件与影响范围

- 触发条件：
  - 存在主对象和流程对象两层结构。
  - 主对象需要做列表筛选、统计或概览展示。
  - 流程本身需要较细的状态机。
- 影响范围：
  - 状态枚举
  - 列表筛选逻辑
  - 明细页展示
  - 状态回写逻辑

## 3. 复现步骤

1. 将完整流程的所有状态直接放到主对象字段中。
2. 让前端使用该字段做列表筛选和概览展示。
3. 随着流程复杂度增加，观察状态变得越来越多且难以解释。
4. 再让子流程明细继续扩展，观察主对象状态进一步失控。

## 4. 根因分析

- 根因结论：
  - 主对象的摘要语义和流程对象的详细语义没有分层建模。
- 直接原因：
  - 一个状态字段同时承担“概览展示”和“流程执行”两种职责。
- 深层原因：
  - 设计阶段没有区分摘要状态与详细状态。
- 为什么此前没有暴露：
  - 在流程较简单时，少量状态还能兼顾两种职责。

## 5. 分步处置方案

1. 把主对象状态收口为摘要状态

```text
主对象状态只保留：
- 列表筛选需要的状态
- 统计需要的状态
- 概览展示需要的状态
```

注意事项：

- 摘要状态数量应尽量少且语义明确。
- 不要把所有节点状态直接暴露给列表层。

2. 把流程对象状态保留为详细状态机

```text
流程对象保留：
- 申请
- 审核
- 驳回
- 撤销
- 完成
- 其它真实业务节点
```

注意事项：

- 详细状态用于流程推进和明细展示。
- 主对象只通过映射规则回写摘要状态。

3. 明确摘要状态与详细状态的映射关系

```text
示例：
- 详细状态 A/B/C -> 摘要状态“处理中”
- 详细状态 D -> 摘要状态“已完成”
```

注意事项：

- 映射关系必须稳定、可解释、前后端一致。
- 不能让前端自己猜测摘要状态。

## 6. 验证与验收标准

- 功能验证点：
  - 主对象列表筛选只依赖摘要状态即可完成。
  - 明细页仍能展示完整流程状态。
- 回归验证点：
  - 新增详细状态不会直接破坏主对象列表逻辑。
  - 前后端对摘要状态含义理解一致。
- 可执行测试用例：
  - 构造多个详细状态，验证映射到主对象后摘要状态正确。
- 通过判定条件：
  - 摘要状态数量可控、语义清晰，流程对象仍保留完整状态机。
