# 状态枚举与状态机一致性规则

## 触发条件
当你在本项目中开发、修改或重构以下场景时，必须优先遵守本规则：

- 订单状态
- 售后状态
- 支付状态
- 提现状态
- 佣金结算状态
- 洗衣卡状态
- 任何包含 `"0"`, `"1"`, `"2"` 这类状态码判断的业务逻辑

## 详细要求

### 1. 固定状态禁止散落魔法值
**关键字**: `状态码` | `魔法值` | `常量`

**内容**:
固定状态值不得在控制器、服务层、前端页面中到处直接写 `"0"`、`"1"`、`"2"`。

**规范**:
- 后端优先使用常量或枚举集中定义
- 前端至少集中定义状态选项、状态文案映射
- 禁止同一状态在不同文件里写出不同文案或不同含义

### 2. 状态流转必须有明确前置条件
**关键字**: `状态机` | `状态跳转` | `非法流转`

**内容**:
每个会变更状态的业务都必须有明确的允许前置状态，不允许任意跳转。

**规范**:
- 审核通过、驳回、完成、取消、退款等动作必须校验当前状态
- 非法状态跳转必须抛出业务异常
- 前端可以按状态隐藏按钮，但后端必须再次校验

### 3. 状态变更与时间字段必须成对
**关键字**: `auditTime` | `finishTime` | `payTime`

**内容**:
状态改变时，应同步补齐对应时间字段和说明字段，不能只改状态值。

**规范**:
- 审核类状态应同步记录 `auditTime`
- 完成类状态应同步记录 `finishTime`、`payTime` 或业务完成时间
- 驳回类状态应同步记录 `rejectReason`
- 使用类状态应同步记录 `useTime`

### 4. 前后端状态文案必须同口径
**关键字**: `tabs` | `el-tag` | `状态文案`

**内容**:
后台 tabs、表格 tag、详情页文案、Swagger 描述与后端状态码含义必须一致。

**规范**:
- 后台页 tabs 选项、tag 文案、详情文案必须复用同一套状态映射
- 不允许出现后端 `0=待审核`、前端却展示成 `未处理` 的漂移情况
- 小程序端和后台端如需使用不同文案，必须确认只是展示差异，不是业务含义差异

### 5. 新增状态前先检查 README 与既有规则
**关键字**: `README` | `业务主线` | `既有状态`

**内容**:
新增或修改状态时，必须先确认 README 中的状态设计和已有规则，不得随意扩展。

**规范**:
- 涉及订单、支付、洗衣卡抵扣：
  - 联动 `order-payment-consistency-rule.md`
- 涉及提现、返佣、佣金：
  - 联动 `finance-audit-flow-rule.md`
- 涉及售后：
  - 结合 README 的售后流程确认状态边界

### 6. 验证清单
每次状态相关逻辑改动后，至少检查：
- 是否仍存在散落魔法值
- 状态动作是否校验了前置状态
- 状态变更是否同步写入时间/原因字段
- 前后端状态文案是否一致
- tabs、tag、详情文案是否与后端口径一致
