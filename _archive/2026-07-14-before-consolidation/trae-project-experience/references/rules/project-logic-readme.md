# 项目核心逻辑获取规则

## 触发条件
全局生效。在处理本项目的任何需求、修复 Bug 或进行代码编写时，如果需要了解或遵循项目的核心逻辑与业务规范，必须遵守此规则。

## 详细要求
### 1. 优先参考 README.md
**内容**:
请务必优先读取并参考项目根目录下的 `./README.md` 文件。

**补充**:
- 读取 README 后，应进一步打开 `.trae/README.md`
- 再根据业务主线打开 `rules/scene-index.md`
- 如果问题同时涉及多个对象、多个状态或多个处理环节，应继续打开 `rules/scene-handling-framework.md`
- 如果问题属于可跨项目复用的通用处置场景，应继续检查 `.trae/kb/maintenance/generated/knowledge-index.json`
- 不允许只看 README 标题后直接编码

### 2. 先按 README 判断所属业务主线
**内容**:
读取 README 后，先判断当前需求属于哪条业务主线，再继续选择对应规则和技能，不允许跳过业务归类直接编码。

**常见归类**:
- 登录、注册、授权、资料补全、归属信息选择、偏好规格选择：身份建立与资料准备主线
- 预览、创建、支付、优惠、地址、时间、资源抵扣、履约启动：交易创建与履约启动主线
- 个人记录、权益、资产、消息、服务申请、进度跟踪：个人资产与服务管理主线
- 后台列表整理、详情展示、筛选治理、主从页面整理：后台管理主线
- 收益、提现、审核、金额回退、结算：资金审核主线

### 3. 读取 README 后必须联动具体规则
**内容**:
不能只停留在“看过 README”，还要根据场景自动联想到对应规则。

**规则映射**:
- 涉及订单预览、下单、支付、洗衣卡抵扣：
  - 必须联动 `order-payment-consistency-rule.md`
  - 同时参考 `status-enum-consistency-rule.md`
- 涉及提现、返佣、佣金、金额回退、审核：
  - 必须联动 `finance-audit-flow-rule.md`
  - 同时参考 `status-enum-consistency-rule.md`
- 涉及库存扣减、库存回退、取消订单释放库存：
  - 必须联动 `product-stock-rule.md`
- 涉及后台列表页整理、tabs、远程用户筛选、详情弹窗：
  - 必须联动 `admin-list-scene-rule.md`
  - 同时参考 `admin-ui-render-rule.md`
  - 同时参考 `project-memories.md`
- 涉及公共查询接口、小程序公共读取能力：
  - 必须联动 `public-controller-reuse-rule.md`
- 涉及列表展示、详情回显、tag 展示、名称展示、摘要展示：
  - 必须先检查是否存在裸外键字段
  - 命中后优先联动 `public-controller-reuse-rule.md`
  - 如果属于通用处置方式，同时联动 `.trae/kb` 中“外键字段优先后端聚合回显业务展示信息”
- 涉及小程序端接口：
  - 必须联动 `mini-program-api-rule.md`
- 涉及状态流转、状态摘要、前后端状态口径统一：
  - 必须联动 `status-enum-consistency-rule.md`

### 4. 优先使用场景索引定位规则
**内容**:
当一个需求可能跨多个主线时，优先打开 `rules/scene-index.md`，按“业务场景 -> 规则 -> 技能”路径继续收敛。

**规范**:
- 后台页面整理类需求，优先从后台管理主线进入
- 订单、支付、洗衣卡抵扣类需求，优先从下单交易主线进入
- 售后、洗衣卡、优惠券、消息类需求，优先从我的资产与服务主线进入
- 接口公开范围、匿名读取、小程序调用边界类需求，优先从公共接口与小程序边界场景进入

### 4.1 通用处置问题必须联动共享知识库
**内容**:
当需求不是单纯的项目业务口径问题，而是“通用处置方式”问题时，必须联动 `rules/kb-bridge-rule.md` 与 `.trae/kb`。

**常见触发**:
- “这种问题以前怎么处理”
- “先看看有没有类似方案”
- “不要从零改，按之前沉淀来”
- “这个问题值不值得沉淀”
- 复杂问题涉及返回结构、状态分层、后端兜底、资源释放、接口分层、列表治理

**规范**:
- 先由当前项目规则判断业务边界
- 再由 `.trae/kb` 提供通用处置模式
- 如果本次新结论具有跨项目复用价值，应建议沉淀到 `.trae/kb`

### 5. 业务约定优先于个人习惯
**内容**:
README 中记录的业务链路、页面职责、状态设计和研发优先级，优先级高于模型默认习惯或泛化实现方式。

**规范**:
- 不得擅自改变 README 已明确的主链路顺序
- 不得把服务型项目按传统商城思维硬套实现
- 不得在未确认前擅自新增 README 未要求的扩展能力

### 6. 验证清单
每次开始编码前至少检查：
- 当前需求属于 README 的哪条主线
- 是否已经查看 `.trae/README.md` 与 `rules/scene-index.md`
- 是否已经联动到对应规则文件
- 是否已经判断该问题需不需要联动 `.trae/kb`
- 是否存在与 README 冲突的实现假设
- 是否把公共、小程序、后台三类接口职责区分清楚
- 如果任务包含渲染、回显、详情展示，是否已经检查裸外键字段并判断是否需要后端聚合回填
