# 后台列表治理规则

## 触发条件
当你在本项目中开发、整理或重构后台管理列表页时，尤其是订单、提现、佣金、合伙人、洗衣卡购买记录、消息公告等页面，必须优先遵守本规则。

## 详细要求

### 1. 状态优先使用 Tabs
**关键字**: `状态 tabs` | `全部` | `列表筛选`

**内容**:
对于订单状态、提现状态、审核状态、支付状态这类强主维度筛选，优先使用页面顶部 `el-tabs`，不要默认做成下拉框。

**规范**:
- `全部` 页签必须使用显式值，如 `all`
- 禁止直接把空字符串 `""` 作为 `el-tabs` 的 `name`
- 页签选中值与接口查询参数分离维护
- 推荐做法：
  - `activeStatus = "all"`
  - `queryParams.status = null`
  - 切换 tabs 时做 `all -> null` 的映射

### 2. 远程用户筛选必须带默认数据
**关键字**: `远程用户筛选` | `默认用户选项` | `el-select`

**内容**:
订单类、资金类、资产类后台页一旦有用户筛选，默认就要提供一批可直接选择的用户选项，不能只支持输入搜索。

**规范**:
- 页面 `created` 时调用一次默认用户选项加载
- 用户下拉展开时，如果当前没有选项，自动补默认数据
- 远程搜索词清空后，恢复默认用户选项，不允许直接置空
- 默认分页建议 `pageNum=1`、`pageSize=20`
- 搜索字段至少支持：用户ID、昵称、姓名、手机号

### 3. 列表字段按场景聚合展示
**关键字**: `聚合列` | `el-table` | `详情摘要`

**内容**:
后台页字段较多时，优先将强关联字段合并到一个列中展示，避免横向平铺。

**规范**:
- 表头仍使用 `align="center"`
- 单元格内部使用 `div` 多行堆叠，内容左对齐
- 推荐聚合维度：
  - 基础信息
  - 金额信息
  - 收款/地址/关联信息
  - 状态与时间
- 详情页优先展示业务字段，不展示 `delFlag`、`createBy`、`updateBy`

### 4. 详情展示优先用 Dialog 或 Drawer
**关键字**: `详情弹窗` | `el-dialog` | `el-drawer` | `el-descriptions`

**内容**:
后台列表页必须具备详情能力。详情展示优先使用 `el-dialog` 或 `el-drawer`，并按业务板块分组展示。

**规范**:
- 优先用 `el-descriptions`、分段标题或内嵌表格展示
- 关联列表数据要分块渲染，不能和主信息混在一列
- 空值统一回显 `--`
- 金额统一格式化两位小数
- 银行卡号、身份证号等敏感信息按业务场景决定是否掩码

### 5. 只读页面必须真正收敛 CRUD
**关键字**: `只读页` | `查询详情` | `收敛 CRUD`

**内容**:
如果页面定位是浏览、审核、记录查看，而不是主数据维护，则必须真实收敛 CRUD，而不只是隐藏按钮。

**规范**:
- 前端移除：`新增`、`修改`、`删除`、`批量操作`、无关提交方法
- API 文件清理不用的增删改方法
- 后端控制器同步移除与只读页不一致的写接口
- 按钮权限、菜单权限、后端权限保持一致

### 6. 列表默认口径要明确
**关键字**: `默认筛选` | `已支付` | `有效记录`

**内容**:
如果某个页面按业务应默认只展示有效数据，必须显式写进页面初始化逻辑，而不是依赖用户手工筛选。

**规范**:
- 例如洗衣卡订单默认只看 `已支付`
- 例如某些审核页默认看 `待审核`
- 默认口径需要同步体现在：
  - tabs 默认选中值
  - 查询参数默认值
  - 重置逻辑

### 7. 验证清单
每次列表治理完成后，至少检查以下内容：

- tabs 是否只有一个激活项
- 远程用户筛选是否有默认选项
- 清空搜索词后是否恢复默认用户数据
- 重置后 tabs 与 queryParams 是否一致
- 列表是否仍有过多无意义字段平铺
- 详情弹窗是否按业务分块展示
- 改动文件是否无前端诊断错误
