# 通用业务场景处理框架

本文档用于把“当前问题”抽象成可复用的通用业务场景，并给出一套不绑定具体功能模块的标准化处理流程。后续凡是具备相似核心特征的需求，都应优先套用本框架完成判断、定位、方案选型和验证。

## 1. 适用目标

本框架适用于以下类型问题：

- 需求不是简单改一个字段，而是涉及完整业务链路判断
- 一个问题会同时影响接口返回、状态流转、页面展示或资源占用
- 用户描述里包含“某个场景应该怎样处理”的业务口径
- 需要先抽象场景，再决定改后端、前端、规则还是数据

本框架不绑定以下具体能力：

- 不限定是订单、售后、洗衣卡、优惠券还是后台页面
- 不限定是小程序端、后台端还是公共接口
- 不限定是新增功能、修复 Bug、整理页面还是统一状态口径

## 2. 场景抽象规则

### 2.1 抽象目标

遇到具体问题时，先把它从“本次临时表述”抽象成“通用业务场景”，只保留以下信息：

- 参与对象：
  - 用户
  - 订单
  - 资源
  - 状态
  - 页面
  - 接口
- 关键动作：
  - 创建
  - 预览
  - 查询
  - 匹配
  - 占用
  - 释放
  - 状态推进
  - 结果展示
- 核心边界：
  - 是否需要前后端同口径
  - 是否存在资源约束
  - 是否存在状态机
  - 是否涉及角色边界
  - 是否涉及是否选择某能力但仍需展示候选结果

### 2.2 需要剥离的个性化细节

以下内容不应直接作为通用场景定义：

- 某一个接口路径名
- 某一个字段名
- 某一个页面文件路径
- 某一次临时数据修正
- 某个业务里独有的按钮文案

### 2.3 通用场景提炼模板

处理时统一写成下面这个模板：

1. 场景名称：
   - 用“对象 + 动作 + 约束/结果”命名
2. 参与对象：
   - 涉及哪些实体、页面、接口、状态
3. 触发条件：
   - 什么输入或前置条件会进入该场景
4. 核心约束：
   - 资源、状态、角色、时间、选择条件
5. 输出期望：
   - 需要返回什么、推进什么、展示什么、阻止什么

## 3. 标准处理流程

### 3.1 模块一：场景识别判断规则

识别阶段必须回答 4 个问题：

1. 这是“链路一致性问题”还是“展示渲染问题”？
2. 这是“单点字段调整”还是“完整场景口径变更”？
3. 这是“已选择后执行”还是“未选择前也要展示候选结果”？
4. 这是“单对象处理”还是“多对象匹配/聚合处理”？

场景归类建议：

- 预览与创建都受影响：
  - 归类为“链路一致性场景”
- 一个对象会匹配多个资源或多个关联对象：
  - 归类为“多对象匹配场景”
- 用户是否勾选某能力会影响结果展示与实际执行：
  - 归类为“选择态与执行态分离场景”
- 页面主要问题是筛选、聚合列、详情结构：
  - 归类为“后台治理场景”
- 问题核心是状态流转、摘要状态、联动状态：
  - 归类为“状态机一致性场景”
- 问题核心是 `/api`、`/public`、`/system` 职责：
  - 归类为“接口边界场景”

### 3.2 模块二：核心问题定位维度

定位阶段必须从以下维度逐项排查：

#### A. 口径维度

- 预览、创建、支付、回调、详情是否共用同一套判断
- 页面展示口径与后端真实处理口径是否一致
- 列表、详情、筛选、统计是否共用同一状态定义

#### B. 对象维度

- 当前返回对象是否足够表达场景
- 是缺字段，还是缺聚合对象，还是缺显式顶层字段
- 是只返回单对象，还是应该返回对象列表

#### C. 状态维度

- 当前状态是详细状态还是摘要状态
- 状态是否存在前后端枚举不一致
- 状态推进、撤销、完成、驳回是否闭环

#### D. 资源维度

- 是否涉及资源占用与释放
- 是否存在并发约束
- 是否需要按优先级分配资源
- 是否允许部分命中、部分失败、部分生效

#### E. 角色与边界维度

- 当前能力应该属于 `/api`、`/public` 还是 `/system`
- 是前端决定行为，还是后端必须兜底
- 是匿名可读、登录可读还是后台可读

### 3.3 模块三：通用解决方案选型逻辑

不同问题，按以下规则选方案：

#### 方案 A：统一服务层口径

适用条件：

- 预览、下单、支付、详情存在同一业务判断
- 用户要求“同步到预览”或“创建后也要一致处理”

选型原则：

- 优先抽到服务层公共构建方法
- 前端不做核心业务判断
- 测试接口复用正式业务逻辑

#### 方案 B：选择态与执行态分离

适用条件：

- 用户未选择某能力时，也需要看到候选信息
- 用户选择后才真正生效

选型原则：

- 返回对象同时表达：
  - 是否可用
  - 候选结果
  - 是否已实际执行
  - 实际生效结果
- “可见”与“执行”不能混成一个布尔值

#### 方案 C：单对象改多对象聚合

适用条件：

- 现有返回只支持单对象，但实际场景会命中多个对象
- 同一单据、页面、查询需要同时展示多个关联结果

选型原则：

- 保留原单对象字段用于兼容旧前端
- 新增对象列表字段表达完整结果
- 单对象字段默认取“优先级最高”的对象

#### 方案 D：状态分层设计

适用条件：

- 主对象只需要摘要状态
- 子对象或流程对象需要详细状态机

选型原则：

- 主对象与子对象分层表达状态
- 摘要状态用于筛选、统计、列表展示
- 详细状态用于流程处理和明细展示

#### 方案 E：后端兜底优先

适用条件：

- 不能依赖前端一定调用下一步接口
- 流程一旦中断会导致数据卡死

选型原则：

- 后端在关键节点增加兜底处理
- 前端可优化体验，但不是唯一保障

### 3.4 模块四：落地验证标准

落地完成后，至少从以下 4 类结果验证：

#### 1. 场景适配校验

- 新框架是否只写死当前功能名
- 是否可迁移到同类“预览/创建/选择/匹配/状态流转”场景
- 是否明确了适用边界和不适用边界

#### 2. 结果正确性校验

- 预览、正式执行、详情查询是否同口径
- 选择前与选择后返回结构是否符合预期
- 单对象与多对象命中场景是否都能覆盖

#### 3. 风险回归校验

- 非目标场景是否被误伤
- 未命中资源时是否能优雅降级
- 前端旧字段是否仍可兼容

#### 4. 可复用性校验

- 是否形成了稳定命名的规则文档
- 是否已接入项目入口规则和场景索引
- 下次出现同类场景时，能否不重复讨论直接套框架

## 4. 场景适配校验规则

一个需求要复用本框架，至少满足以下 3 条中的 2 条：

- 同时涉及“判断 + 返回 + 执行 + 验证”中的两个以上环节
- 同一问题可被抽象为多个对象之间的匹配、聚合、状态流转或边界控制
- 处理方式可迁移到除当前业务外的其它模块

以下情况不建议强行套用本框架：

- 纯文案修改
- 单一字段更名
- 单页面纯样式调整
- 一次性数据修复且没有稳定业务规则

## 5. 推荐输出模板

以后处理同类问题时，建议按以下顺序输出：

1. 先给出场景抽象结果
2. 再说明属于哪类通用场景
3. 再给出按四大模块拆解后的处理流程
4. 最后给出本次落地到项目中的具体修改点

## 6. 与本项目规则的联动方式

本框架是所有场景规则的上层方法论，使用顺序如下：

1. 先读 `project-logic-readme.md`
2. 再读 `scene-index.md`
3. 如果需求复杂，先套用本框架完成场景抽象
4. 再进入订单、后台、资金、公共接口等具体规则
