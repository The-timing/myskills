# 主动提取原型需求

## 核心原则

原型中的页面树、批注、黄色标记、显隐条件、状态和交互均为需求来源。必须主动提取并回写需求／开发文档，不能等待用户逐条指出。

## 执行流程

1. 收集可用原型证据：原型链接、截图、页面树、批注、需求文档与现有工程。
2. 逐页检查页面名称、入口、按钮、字段、状态、跳转、默认值和异常提示。
3. 单独检查所有批注、黄色说明、红箭头、图层备注和页面树中的隐藏页面；它们优先于对页面视觉的推测。
4. 将每条需求归入唯一职责域：
   - **顾客端：** 平板／手机页面、可见入口、交互、公开接口。
   - **管理后台：** 内容维护、菜单、运营配置、统计、权限。
   - **服务端与数据：** 第三方调用、状态流转、数据表、删除、定时任务和安全。
5. 立即将已确认事项写入对应文档章节；无法确认的事项列入「待确认项」，不得自行编造。
6. 完成后执行「原型证据 → 文档章节」对照，确认每条批注都有落点或待确认记录。

## 映射规则

| 原型信息 | 必须落入的位置 |
| --- | --- |
| 顾客端入口、显隐、页面流程 | 顾客端功能与交互规则 |
| 「后台可控制」「可配置」「开关」 | 后台菜单、配置归属、权限和服务端配置来源 |
| 统计、列表、删除、编辑 | 后台业务页面与后台规则 |
| 接口、扫码、上传、回调、定时删除 | 服务端与数据规则；仅顾客端公开接口列入顾客端 API |
| 标注不清、接口字段未知 | 待确认项，说明阻塞内容 |

## 强制检查

- 不要因原型页面无法渲染或无法自动读取就跳过批注；继续检查用户提供的截图、页面树和文字说明。
- 不要把后台配置写进顾客端功能，也不要将后台管理接口混入顾客端 API 清单。
- 不要仅写「支持配置」；明确配置入口、配置项、默认值、权限、下发位置和生效时机。
- 顾客端显隐规则必须同时写明后台控制位置。例如「后台可控制仅展示测肤」必须同时体现为：顾客端侧边栏仅显示测肤入口、后台侧边栏配置可关闭其他入口、配置通过首页数据下发。
- 每次用户补充截图或原型说明时，重新执行受影响页面和批注的对照检查，并主动更新文档。

## 交付标准

交付时简要说明：已提取的原型要点、对应文档位置、仍待确认的事项。不要只回答「已整理」或等待用户继续指出遗漏。
